# CBSA Engine — Tech Explanation Brief
## Continuous Behavioral Streaming Authentication

> **Purpose of this document:** A clear, justified explanation of the CBSA behavioral engine. Every design decision is accompanied by (a) the problem it solves, (b) alternatives that were rejected and why, and (c) a factual/mathematical basis for the choice.

---

## 1. What Problem Are We Solving?

Traditional authentication (passwords, OTPs, PINs) is **episodic** — it authenticates the user once at login but does nothing after. Once a session is open, an attacker who steals that session (via session hijacking, token theft, or shoulder-surfing) operates with full authenticated access for the remainder of the session.

**CBSA solves this by authenticating continuously** — every interaction event from the mobile app is analyzed in real time and contributes to a live trust score. If behavior deviates from the enrolled user's profile at any point in the session, trust drops and the system triggers a challenge or rejection.

**Why is this needed?**
- OWASP places Broken Authentication in its Top 10 vulnerabilities.
- 61% of data breaches involve credential-based attacks (Verizon DBIR 2023).
- Mobile banking apps are particularly exposed: a stolen phone with a live session grants full access.
- Behavioral biometrics operate passively — no additional burden on the legitimate user.

## 3. The 48-Dimensional Behavioral Vector

Every mobile interaction is encoded as a **48-dimensional feature vector** — each dimension is a normalized `[0,1]` measurement of how the user interacts:

- Touch pressure and contact area
- Scroll velocity and deceleration
- Typing inter-key intervals (digraph timing)
- Gesture trajectory curvature
- Tap/hold duration ratios

**Why 48 dimensions?**
48 is the agreed mobile SDK interface. 48 features strike a balance: enough dimensions to capture behavioral identity discriminatively (high entropy), few enough to avoid the curse of dimensionality in distance computations.

**Why normalize to `[0,1]`?**
All similarity, drift, and stability computations downstream assume a bounded input space. Out-of-range values break mathematical guarantees — for example, Mahalanobis distance amplifies out-of-range values when variance is small, producing unbounded similarity scores. Layer 1 enforces this contract at the entry point.

## 4. Layer 1 — Why We Validate at Ingestion

### Nonce Uniqueness Check

**Problem:** Without nonce checking, a replay attack can re-inject a previously captured authentic behavioral event. Repeated real events inflate similarity scores, allowing an attacker to replay a captured session and maintain high trust.

**Solution:** Each event carries a cryptographic nonce generated by the mobile SDK. If a nonce is seen twice, the event is rejected immediately. No state is modified on rejection.

### Burst-Rate Guard

**Problem:** Automated event injection (fuzzing, scripted replay) can submit events at sub-human speed.

**Solution:** Events arriving faster than 40ms inter-event intervals, sustained for 5+ consecutive events, are rejected. **Why 40ms?** The practical lower bound of human touch latency — below this, interaction cannot be human. *(Reference: MacKenzie & Buxton, 1992, on minimum human tapping intervals.)*

## 5. Layer 2a — Three-Scale Temporal Drift

### The Core Idea

Behavioral drift measures: **how far is the current event from where the user usually is?**

The key insight is that "usually" is not a single timescale. Behavioral identity has structure at multiple temporal resolutions:

| Scale | Window | Captures |
|-------|--------|----------|
| **Short** | 5 events (~5s) | Micro-behavioral: sudden impostor switches |
| **Medium** | 20 events (~20s) | Episodic: context transitions, gradual impersonation |
| **Long** | Full session (Welford) | Identity baseline: slow multi-event drift |

**Why three scales instead of one?**

A single short window is blind to gradual drift. A single long window is too slow to catch sudden takeovers. A single medium window misses both extremes.

*Specific threat each scale catches:*
- **Short-only attacker:** Someone who copies the user's typing style but at a different pace. Caught by medium/long.
- **Gradual attacker:** Someone who slowly morphs behavior over 50+ events. Caught by long drift.
- **Sudden hijack:** An attacker who picks up a device mid-session. Caught immediately by short drift.

**The three-scale system catches all three at zero additional computational cost** — the windows are maintained anyway in session state.

### The Leakage Problem and Fix

**What is leakage?**
If we compute drift of event v_t against statistics that already include v_t, the event appears in both sides of the comparison. In the extreme case (session's first event), the running mean equals v_t exactly → drift is always zero → completely uninformative.

**Fix:** We atomically capture a pre-update snapshot *before* incorporating v_t, then compute all drift metrics against only that snapshot. v_t never appears in its own reference distribution.

```
Pre-update snapshot: { mean, variance, windows } ← before v_t is added
Update buffer with v_t
Compute all drift metrics using ONLY the snapshot
```

This is a fundamental correctness requirement, not an optimization.

### Why Exponential Normalization?

Raw L2 distance is unbounded and scale-dependent. We need a bounded drift signal.

Three candidates:

| Method | Problem | Chosen? |
|--------|---------|---------|
| `min(d / d_max, 1)` — clipping | Requires choosing d_max without principled basis; clips real outliers | No |
| `d / (1 + d)` — logistic | No natural scale parameter; insensitive at high d | No |
| `1 - exp(-d / σ)` — exponential | Natural scale parameter σ; smooth; derivative strictly decreasing | **Yes** |

**Why exponential wins:** σ = 0.15√D is grounded in reality — a "moderate" behavioral change shifts each dimension by 0.15 on average. At exactly that deviation, drift scores `1 - 1/e ≈ 0.63` (the one-sigma point). Thresholds have a principled, interpretable unit — they are measured in units of "moderate behavioral change."

### Welford's Algorithm for Running Statistics

We maintain an online running mean and variance using Welford's algorithm instead of the naive formula `Σx²/n - (Σx/n)²`.

**Why?** The naive formula suffers catastrophic cancellation — the two terms become nearly equal for large n, and their difference loses floating-point precision. For sessions of up to 10,000 events with 48-dimensional vectors, this is a practical accuracy problem, not a theoretical one.

*Reference: Welford, B.P. (1962). "Note on a method for calculating corrected sums of squares and products." Technometrics, 4(3), 419–420.*

## 6. Layer 2b — Prototype Engine

### What is a Prototype?

A **prototype** is a stored behavioral archetype for the user — a 48-D centroid vector representing one characteristic behavioral mode. A user may have multiple prototypes (e.g., one for typing-heavy tasks, one for browsing-only tasks).

For each incoming event, we compute how similar the current behavioral window is to the best-matching prototype.

### Composite Similarity Formula

```
sim = 0.50 × cos(v, μ) + 0.40 × exp(-d_M/√D) + 0.10 × S
```

Three geometrically orthogonal components:

| Component | Weight | Captures | Why this weight |
|-----------|--------|----------|-----------------|
| **Cosine similarity** | 0.50 | Behavioral direction / shape — scale-invariant | Primary identity signal; two vectors of different magnitude but same behavioral pattern have cosine = 1 |
| **Mahalanobis kernel** | 0.40 | Magnitude deviation, personalized by variance | Same deviation is penalized more for a low-variance user — consistent with individual expectations |
| **Stability modifier** | 0.10 | Coherence quality of current window | A coherent window should contribute slightly more; minor adjustment, not an identity signal itself |

**Why cosine + Mahalanobis together?**
- Cosine is scale-invariant but magnitude-blind.
- Mahalanobis is sensitive to magnitude but not to direction alone.
- Together they cover both what-direction and how-far — two independent aspects of behavioral identity.

**Why diagonal (not full) Mahalanobis?**

Full covariance estimation for D=48 requires at minimum O(D²) = 2,304 samples for the matrix to be non-singular. Sessions have 20–100 events per prototype — full covariance is impractical and numerically unstable. Diagonal approximation:
- Requires only D samples (easily satisfied)
- Captures per-feature variance accurately (touch pressure and scroll velocity have very different natural variances)
- Is exact when features are uncorrelated — a reasonable approximation for well-engineered behavioral features

### Prototype Quarantine — Why We Gate New Prototypes

**Problem:** Without a creation gate, an attacker with brief physical access can inject one event and create a new prototype. On the next access, the attacker's behavior matches their own injected prototype → high similarity → system is compromised.

**Solution:** Candidate vectors must satisfy all three conditions before a prototype is created:

| Condition | Value | Rationale |
|-----------|-------|-----------|
| Minimum 3 observations | `N_MIN = 3` | One event: trivial noise. Two events: insufficient for 48-D centroid. Three events: minimum non-trivial estimate |
| Spread over 30+ seconds | `T_MIN = 30s` | Human behavior emerges over minutes, not seconds. 30s defeats burst-injection (attacker cannot satisfy temporal spread by rapid-firing events) |
| Directional consistency ≥ 0.72 cosine | `CONSISTENCY = 0.72` | In 48-D space, cosine 0.72 = angle ≤ 44°. All observations must lie within a 44° cone — they form a coherent behavioral cluster, not scattered noise |

**Security consequence:** An attacker who gains momentary physical access needs at minimum 3 events spread over 30+ seconds with consistent behavioral patterns — a substantially higher bar than O(1) event injection.

### Adaptive Learning Rate

Prototype updates use EMA with an exponentially decaying learning rate:

```
η(n) = η_base × exp(-n/τ) + η_floor        (η_base=0.30, τ=50, η_floor=0.01)
```

| Stage | n | η | Effect |
|-------|---|---|--------|
| New prototype | 1 | 0.30 | Fast calibration from limited data |
| Maturing | 50 | 0.12 | Slowing down |
| Mature | 100 | 0.011 | Resistant to noise |
| Asymptotic | ∞ | 0.01 | Never freezes — accommodates genuine long-term change |

**Why a floor?** Without η_floor, a mature prototype would eventually stop updating. But real users change habits over months. η_floor = 0.01 means even a prototype with n=10,000 still adapts 1% toward each new observation, allowing genuine behavioral evolution to propagate through the model without allowing noise to corrupt it.

### Quality-Based Prototype Lifecycle

When the maximum prototype count is reached (MAX_PROTOTYPES = 15), the lowest-quality prototype is removed:

```
Q(k) = log(1 + n_k) × exp(-λ_age × age_k) × max(sim_k, 0.1)
```

| Component | Effect |
|-----------|--------|
| `log(1+n)` | Support matters, but logarithmically — 100 observations is valuable, not 10× more than 10 |
| `exp(-λ·age)` | A prototype not matched in 1 day scores 37% of its original Q; after 7 days: 0.1% |
| `max(sim, 0.1)` | A prototype not matching current behavior scores lower; min 0.1 prevents zero |

This keeps the prototype set tuned to the user's *current* behavioral state, not their historical one.

## 7. Layer 2c — Behavioral Session Fingerprint (Markov Transition Surprise)

### The Gap in Existing Signals

Drift, stability, and similarity all measure **what** the user's behavioral vectors look like — their magnitude, shape, and position in 48-D space. None of them measure **order** — whether the user navigates the app in their habitual sequence.

**The threat this closes:** An attacker who has observed or recorded the genuine user's feature vectors can reproduce the magnitude and shape, but is likely to navigate the app in an unexpected order. For example:
- Genuine user: always checks account balance → reviews recent transactions → initiates transfer
- Attacker: goes straight to transfer screen (skipping the habitual preamble)

All three existing signals would see "transfer screen event vectors" — correct. Only the transition engine asks: "but did you arrive here via the expected navigation path?"

### How It Works

Each user builds a **Markov transition probability matrix** — a table of "when I came from event type A, how often did I go to event type B?"

When a new event arrives, we check how probable the current transition is:

```
I(prev→curr)        = -log₂( P[prev→curr] )      (Shannon information in bits)
transition_surprise = 1 - exp(-I / σ)              (normalized to [0,1))
```

**Intuition:**

| Scenario | P[prev→curr] | Surprise | Meaning |
|----------|-------------|---------|---------|
| Always-expected transition | ≈ 1.0 | ≈ 0.0 | Habitual path — full trust contribution |
| Seen half the time | 0.5 | 0.28 | Somewhat unusual — mild penalization |
| Atypical but possible | 0.1 | 0.67 | Unusual navigation — moderate penalization |
| Never seen before | → 0 (floored) | ≈ 0.92 | Completely novel path — strong penalization |

### Why Shannon Information Content?

Shannon's information content `I = -log₂(P)` is the mathematically correct measure of "how much information does observing this event carry?" — equivalently, "how surprised should you be?"

- `I = 0` when P = 1 (no surprise — fully expected)
- `I = 1 bit` when P = 0.5 (binary uncertainty)
- `I = 3.32 bits` when P = 0.1 (one-in-ten chance)
- `I → ∞` as P → 0

This is not an arbitrary formula — it is the unique measure of surprise consistent with information theory's axioms *(Shannon, 1948, "A Mathematical Theory of Communication")*.

The exponential normalization `1 - exp(-I/σ)` then maps bits of surprise to `[0,1)` with σ = 3.0 bits as the normalization scale (at 3 bits of surprise, the score is 0.63 — "notably unusual but not at maximum alert").

### EMA Matrix Update — Learning Navigation Habits

The probability matrix is updated after each event using EMA:

```
For all k:   P[A][k] *= (1 - α)       ← decay all entries  (α = 0.15)
             P[A][B] += α              ← reinforce observed transition
```

This means:
- Frequently made transitions accumulate weight over time
- Transitions not made recently gradually fade
- The matrix adapts to evolving navigation habits without requiring a fixed history window

**Why α = 0.15?** At α = 0.15, a transition seen 10 consecutive times reaches ~80% of its long-run EMA weight. This provides responsive learning while requiring repeated evidence before a transition becomes "expected."

### Leakage-Free Property

The same anti-leakage discipline applied to drift is applied here:

```
Step 1: Read P[prev→curr]   ← pre-update (before this transition is incorporated)
Step 2: Compute surprise     ← using pre-update probability
Step 3: Update P with A→B   ← post-scoring update
Step 4: Advance prev_event  ← chain advance for next event
```

If step 3 came before step 1, the current transition would inflate its own probability, systematically under-estimating the surprise of novel transitions and hiding sequential anomalies.

### Why This is Orthogonal to All Other Signals

| Signal | What it measures | What it ignores |
|--------|-----------------|-----------------|
| Drift | How far the vector is from prior means | Order of arrivals |
| Stability | Coherence within a window | Which page was visited |
| Similarity | Shape match to stored prototype | Navigation sequence |
| **Transition surprise** | Navigation order vs. habit | Feature vector values |

The transition engine provides strictly complementary information. An attacker can beat any of the first three by copying feature vectors. They cannot beat the transition engine without also replicating the user's exact navigation sequence — a substantially harder attack.

## 8. Layer 4 — Trust Engine

### Raw Trust Signal

All behavioral signals are combined into a single raw trust value:

```
R_t = 0.40 × sim + 0.20 × S + 0.30 × (1 - D_t) + 0.10 × (1 - ts_t)

D_t = 0.50 × d_short + 0.30 × d_medium + 0.20 × d_long
```

| Term | Weight | Rationale |
|------|--------|-----------|
| Prototype similarity | 0.40 | Primary identity signal |
| Stability | 0.20 | Behavioral coherence |
| Drift-complement | 0.30 | Penalizes deviation at all timescales |
| Transition familiarity | 0.10 | Sequential navigation match |

**Bounds by construction:** Maximum = 0.40+0.20+0.30+0.10 = 1.0. Minimum = 0. R_t ∈ [0,1] without any clipping.

**Why `(1 - ts_t)` and `(1 - D_t)`?** Both drift and surprise are "badness" signals. Inverting them gives "goodness" signals so all four terms contribute positively to trust. A familiar transition → ts=0 → (1-ts)=1 → full contribution. An unexpected transition → ts≈1 → (1-ts)≈0 → no contribution.

### EMA Trust Model

The raw signal is not the trust score directly — it feeds an Exponential Moving Average:

```
T_t = α_t × T_{t-1} + (1 - α_t) × R_t
```

**Why EMA?** A single anomalous event should not immediately drop trust for an established session — natural behavioral noise would cause constant false rejections. The EMA provides temporal inertia: trust changes gradually for stable sessions and quickly for sustained anomalies.

### Adaptive Alpha — Two Modulations

The EMA coefficient α is not constant. It is modulated by two factors:

**1. Short drift (event-by-event):**
```
α_t = clip(α_eff_max - γ × d_short, α_min, α_eff_max)      γ = 0.55
```

- Stable behavior → α near 0.85 → half-life ~4 events → strong inertia
- Sudden anomaly → α near 0.30 → half-life ~0.57 events → fast response

**Why?** High inertia is appropriate for trusted, stable sessions. But when an impostor takes over, we need the trust score to drop within 1–2 events, not over 5–10. The drift-dependent alpha gives us both — stability when behavior is stable, speed when it is not.

**2. Prototype topology cohesion (user-level):**
```
α_eff_max = ALPHA_MAX × (0.90 + 0.10 × cohesion)
```

Cohesion measures how geometrically aligned the user's stored prototypes are. A user with cohesion = 1 has a single consistent behavioral mode; a user with low cohesion has genuinely different patterns in different contexts.

- High cohesion → full inertia appropriate (single mode, trust should be stable)
- Low cohesion → slightly reduced inertia (multi-modal user, trust needs to respond to context switches)

### Trust Zones and Hysteresis

| Zone | Range | Decision |
|------|-------|----------|
| SAFE | [0.80, 1.0] | ACCEPT |
| MONITOR | [0.40, 0.80) | LOG only |
| RISK | [0.0, 0.40) | REJECT |

**Why a wide MONITOR zone?** Without a dead zone, trust scores oscillating near a threshold produce alternating ACCEPT/REJECT decisions — an annoying and insecure outcome for legitimate sessions with natural behavioral variance. The 0.40–0.80 range provides hysteresis: once trust is established, minor variations do not cause disruption.

## 9. What Makes This Defensible as a Research System

| Property | How We Guarantee It |
|----------|---------------------|
| All outputs bounded [0,1] | Mathematical bounds proven for every component; runtime `InvariantError` raised on any violation |
| No data leakage | Pre-update snapshots; atomically enforced in `update_session_buffer()` |
| Interpretable signals | Every signal has a named meaning and unit (bits, normalized L2, cosine, variance ratio) |
| Adaptive to users | EMA with decaying η; prototype lifecycle; cohesion-modulated alpha |
| Resistant to cold-start attacks | Quarantine gate; 20-event warmup before any trust decisions |
| Resistant to replay attacks | Layer 1 nonce uniqueness; burst-rate guard |
| Multi-timescale coverage | Short/medium/long drift hierarchy; no temporal blind spot |
| Sequential attack coverage | Markov transition fingerprint catches the gap left by all spatial signals |
| Computationally lightweight | All Layer 2 operations are single-pass over a bounded buffer; no matrix inversions; no neural network in the hot path |

## 10. Summary in One Paragraph

CBSA continuously authenticates a mobile banking user by analyzing every interaction event from the app. Layer 1 validates events and blocks replays. Layer 2 extracts four families of behavioral signals: three-scale temporal drift (how far is current behavior from the user's short, medium, and long-term baseline), behavioral stability (how coherent is the current interaction window), prototype similarity (how well does current behavior match stored behavioral archetypes), and Markov transition surprise (is the user navigating in their habitual sequence?). Layer 4 combines all signals into a continuous trust score via an adaptive EMA model that provides strong temporal inertia for stable sessions and fast response to sudden behavioral anomalies, issuing SAFE, MONITOR, or RISK decisions. Every intermediate signal is mathematically bounded, leakage-free, and invariant-checked at pipeline boundaries.
